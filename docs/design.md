# AI 小说转剧本工具设计文档

## 1. 项目概述

项目暂定名：Story2Script AI。

Story2Script AI 是一款面向小说作者的 AI 辅助剧本创作 Web 工作台。用户导入 3 个章节以上的小说文本后，系统会自动完成章节分析、人物与地点整理、剧情节点抽取，并生成符合 YAML Schema 的结构化剧本初稿。作者可以继续编辑 YAML、查看剧本预览、运行结构校验，并导出可进一步打磨的剧本文档。

本项目的核心目标不是让 AI 输出一段不可控的文本，而是把小说改编过程拆成可理解、可校验、可编辑的结构化工作流，降低小说作者从叙事文本进入剧本创作的门槛。

## 2. 题目理解

比赛题目原文要点：

- 能将 3 个章节以上的小说文本自动转换为结构化剧本。
- 输出格式必须为 YAML。
- 目标是“可编辑、可进一步打磨的剧本初稿”。
- 核心价值是“降低改编门槛，提升效率”。
- 需要额外编写一篇 YAML Schema 文档，并说明该 Schema 的设计原因。

对题目的深层理解（直接决定优先级排布）：

1. **“降低门槛、提升效率”是题眼。** 不只是“能转”，而是要让不熟悉剧本格式的作者用得顺。因此“可编辑 + 实时校验 + 可读预览”不是加分项，而是基本盘，不可裁剪。
2. **“3 个章节以上”是硬性验收点。** 系统必须能识别章节数量，少于 3 章要拦截并提示，且内置样例必须 ≥3 章。这是评委最容易第一时间验证的点。
3. **Schema 文档与设计原因是独立评分项。** 题目单独点名，必须与代码同等重视、保持一致，不能留到最后赶工。
4. **“可进一步打磨的初稿”意味着结果可带走。** 必须支持导出 YAML（结构化继续编辑）与可读剧本预览/导出（分享与二次创作）。

因此本项目的验收重点不是单纯“生成得像剧本”，而是：

- 输入链路完整：用户能导入多章节小说，且对章节数量有硬性校验。
- 改编链路完整：系统能理解章节、人物、地点、事件和冲突。
- 输出链路完整：生成的剧本是结构化 YAML，而不是普通自然语言。
- 编辑链路完整：作者可以继续修改和导出。
- 工程链路完整：YAML 可以被 Schema 校验，错误可以被定位和修复。

## 3. 用户与使用场景

目标用户包括：

- 想把自己小说改成短剧、影视剧或广播剧的网文作者。
- 缺少专业剧本格式经验，但希望快速得到剧本初稿的个人创作者。
- 想将已有故事 IP 做成视频脚本、分场大纲或短剧提案的内容团队。

典型使用场景：

1. 作者粘贴或上传 3 个以上章节的小说。
2. 系统识别章节，并展示每章摘要。
3. 作者选择改编类型与改编强度。
4. 系统生成剧本 YAML。
5. 作者在 YAML 编辑器中修改内容。
6. 系统实时校验 YAML，并渲染剧本预览。
7. 作者导出 YAML 或 Markdown 版本，继续二次创作。

## 4. 产品设计目标

### 4.1 完整度目标

- 支持 3 个章节以上小说输入。
- 支持自动切章与章节确认。
- 支持多阶段 AI 改编流程。
- 支持 YAML 剧本输出。
- 支持 YAML 编辑、校验、预览和导出。
- 提供 YAML Schema 设计文档。
- 提供可直接演示的内置样例。

### 4.2 创新性目标

本项目的创新点集中在“结构化改编”而不是“聊天式生成”：

- 使用多阶段 AI 流水线，将长文本改编拆成章节分析、剧本圣经、分场生成和结构修复。
- 使用 YAML Schema 约束输出，保证剧本可编辑、可校验、可扩展。
- 每个场景保留 `source_chapters` 和 `original_reference`，让作者知道 AI 改编依据来自哪里。
- 引入 `purpose` 和 `conflict` 字段，让每个场景具有明确戏剧功能。
- 将 YAML 渲染成可读剧本预览，降低非技术作者使用 YAML 的门槛。

### 4.3 工程质量目标

- 前后端模块边界清晰。
- AI 调用通过 provider 层封装，便于替换模型。
- YAML 解析和 Schema 校验在本地完成，减少不可控输出。
- 生成结果失败时有明确错误路径，而不是只提示“生成失败”。
- 代码提交按功能拆分，避免最后一次性大提交。

### 4.4 演示表达目标

Demo 需要在 3 分钟内跑通：

1. 选择内置小说样例。
2. 展示章节识别结果。
3. 点击生成剧本。
4. 展示 YAML、Schema 校验和剧本预览。
5. 故意修改一个 YAML 字段，展示校验错误。
6. 导出 YAML 或 Markdown。

## 5. 核心功能

### 5.1 小说导入

输入方式：

- 粘贴纯文本。
- 上传 `.txt` 或 `.md` 文件。
- 使用内置示例小说。

输入限制：

- 至少包含 3 个章节。
- 单章内容允许为空时必须提示用户补全。
- 如果无法自动切章，需要允许用户使用分隔符重新切分。

### 5.2 自动切章

优先识别以下章节格式：

- `第1章`、`第一章`、`第 1 章`
- `Chapter 1`
- Markdown 标题，例如 `# 第一章`、`## 第一章`
- 用户手动插入的分隔符，例如 `---chapter---`

切章结果需要展示：

- 章节标题。
- 章节顺序。
- 字数。
- 内容预览。

### 5.3 章节分析

系统对每个章节生成结构化分析：

- 章节摘要。
- 出场人物。
- 重要地点。
- 关键事件。
- 主要冲突。
- 可改编为剧本的动作、对白或悬念点。

章节分析的目标是给后续剧本生成提供稳定中间结果，避免一次性让模型处理全文导致格式失控。

### 5.4 剧本圣经

剧本圣经是全局改编上下文，包括：

- 故事一句话简介。
- 主题。
- 主要人物表。
- 人物关系。
- 地点表。
- 时间线。
- 主线矛盾。
- 改编原则。

剧本圣经用于解决跨章节一致性问题，例如人物名称不统一、地点重复、事件顺序混乱。

### 5.5 YAML 剧本生成

生成结果必须符合 `docs/yaml-schema.md` 定义的 Schema。

最小剧本结构包括：

- 项目信息。
- 来源章节。
- 人物表。
- 地点表。
- 分场剧本。
- 每场的动作、对白、旁白、转场或内心独白。
- 每场的改编原因与原文依据。

### 5.6 YAML 编辑器

编辑器能力：

- 展示生成后的 YAML。
- 支持用户直接修改。
- 修改后可以重新校验。
- 校验通过后可以更新预览。

推荐使用 Monaco Editor 或 CodeMirror。

### 5.7 Schema 校验

校验分为两层：

1. 结构校验：字段类型、必填字段、枚举值、数组结构。由 JSON Schema（`ajv`）完成。
2. 引用与一致性校验：在应用层完成，JSON Schema 无法表达的跨字段规则放在这里，包括：
   - `scene.characters`、`dialogue.speaker`、`location_id`、`source_chapters`、`relationships.target` 等引用必须指向已定义对象。
   - **条件必填**：当 `beat.type` 为 `dialogue` 或 `inner_voice` 时，`speaker` 必填且必须存在。该规则在应用层实现，不强行塞进 JSON Schema。
   - **ID 唯一**：`characters`、`locations`、`scenes` 内各自的 `id` 必须唯一（硬校验，预览排序依赖它）。
   - **章节覆盖**：每个 `source.chapters[].id` 至少被一个 `scene.source_chapters` 引用，否则提示“该章节未生成任何场景”，避免 AI 漏掉章节。

> 说明：条件必填、ID 唯一、章节覆盖这三条属于跨字段语义校验，标准 JSON Schema 表达困难或冗长，统一下沉到应用层校验器，职责更清晰。

错误展示示例：

```text
scenes[2].beats[1].speaker: 未在 characters 中定义
scenes[0].location_id: 未在 locations 中定义
project.title: 必填字段缺失
```

### 5.8 剧本预览

系统将 YAML 渲染为更接近创作者阅读习惯的剧本文本：

```text
第 1 场 雨夜归来

地点：旧火车站
时间：夜晚
人物：林舟、沈念

雨水砸在站台铁皮棚上，林舟拖着旧行李箱走出车厢。

沈念：
你还是回来了。

林舟：
只是回来办点事。
```

### 5.9 导出

支持导出：

- `.yaml`：结构化剧本，可继续编辑或进入其他工作流。
- `.md`：可读剧本预览，方便作者分享。

## 6. AI 多阶段流水线

本项目采用多阶段 AI 流水线，避免一次性生成导致长文本丢失、YAML 损坏、人物不一致。

### 6.1 阶段 1：章节切分与清洗

输入：原始小说文本。

输出：

- 章节数组。
- 章节标题。
- 章节顺序。
- 清洗后的章节正文。

### 6.2 阶段 2：逐章分析

输入：单个章节。

输出：

- 摘要。
- 人物。
- 地点。
- 事件。
- 冲突。
- 改编建议。

### 6.3 阶段 3：生成剧本圣经

输入：全部章节分析结果。

输出：

- 统一人物表。
- 统一地点表。
- 主题与主线。
- 人物关系。
- 时间线。

### 6.4 阶段 4：生成分场剧本 YAML

输入：

- 章节分析结果。
- 剧本圣经。
- 改编类型。
- 改编强度。
- YAML Schema 摘要。

输出：

- 完整剧本 YAML。

### 6.5 阶段 5：本地 Schema 校验

输入：模型生成的 YAML。

输出：

- 校验是否通过。
- 错误路径。
- 错误原因。

### 6.6 阶段 6：结构修复

如果 YAML 校验失败，将错误信息和原 YAML 一起发送给模型，只允许模型修复结构问题，不允许重写剧情。

修复 prompt 的原则：

- 只返回 YAML。
- 不新增无依据剧情。
- 保留已有人物、地点、场景。
- 修复缺失字段、错误引用、枚举值错误和缩进问题。

## 7. 系统架构

### 7.1 前端

推荐技术栈：

- React
- Vite
- TypeScript
- Tailwind CSS
- Monaco Editor 或 CodeMirror
- `js-yaml`

主要页面区域：

- 小说输入区。
- 章节确认区。
- 生成控制区。
- YAML 编辑区。
- Schema 校验区。
- 剧本预览区。

### 7.2 后端

推荐技术栈：

- Node.js
- Express 或 Fastify
- TypeScript
- OpenAI-compatible LLM provider
- `ajv`
- `js-yaml`

后端职责：

- 接收文本和生成参数。
- 调用 LLM。
- 执行 YAML 解析与 Schema 校验。
- 返回结构化错误。
- 提供导出内容。

**Provider 层必须同时提供 `MockProvider` 与 `OpenAIProvider`，通过环境变量切换。** `MockProvider` 返回写死的合法 YAML，不调用真实 API。这样在断网、API 限流、欠费或现场不可控时，demo 主流程依然能完整跑通——这是比赛现场最常见的翻车原因，必须提前兜底。

### 7.3 数据存储

MVP 不强依赖数据库。

优先使用：

- 浏览器状态保存当前项目。
- LocalStorage 保存最近一次编辑内容。
- 导出文件保存最终结果。

如果后续扩展项目管理，可引入 SQLite 或轻量服务端存储。

## 8. API 设计

### 8.1 章节切分

```http
POST /api/chapters/split
```

请求：

```json
{
  "text": "第一章..."
}
```

响应：

```json
{
  "chapters": [
    {
      "id": "chapter_001",
      "title": "第一章 雨夜归来",
      "order": 1,
      "content": "...",
      "word_count": 2600
    }
  ]
}
```

### 8.2 章节分析

```http
POST /api/chapters/analyze
```

请求：

```json
{
  "chapters": [
    {
      "id": "chapter_001",
      "title": "第一章 雨夜归来",
      "order": 1,
      "content": "..."
    }
  ]
}
```

响应：

```json
{
  "analyses": [
    {
      "chapter_id": "chapter_001",
      "summary": "...",
      "characters": ["林舟", "沈念"],
      "locations": ["旧火车站"],
      "key_events": ["林舟回到故乡"],
      "conflicts": ["林舟隐藏回乡原因，沈念察觉异常"],
      "adaptation_notes": ["旧信可以作为视觉悬念道具"]
    }
  ]
}
```

### 8.3 生成剧本

```http
POST /api/screenplay/generate
```

请求：

```json
{
  "chapters": [],
  "analyses": [],
  "adaptation_type": "screenplay",
  "adaptation_intensity": "balanced"
}
```

响应：

```json
{
  "yaml": "schema_version: \"1.0\"...",
  "validation": {
    "valid": true,
    "errors": []
  }
}
```

### 8.4 YAML 校验

```http
POST /api/yaml/validate
```

请求：

```json
{
  "yaml": "schema_version: \"1.0\"..."
}
```

响应：

```json
{
  "valid": false,
  "errors": [
    {
      "path": "scenes[0].beats[1].speaker",
      "message": "speaker 未在 characters 中定义"
    }
  ]
}
```

### 8.5 YAML 修复

```http
POST /api/yaml/repair
```

请求：

```json
{
  "yaml": "schema_version: \"1.0\"...",
  "errors": [
    {
      "path": "project.title",
      "message": "必填字段缺失"
    }
  ]
}
```

响应：

```json
{
  "yaml": "schema_version: \"1.0\"...",
  "validation": {
    "valid": true,
    "errors": []
  }
}
```

## 9. 评审规则对齐

### 9.1 作品完整度与创新性 40%

对应设计：

- 完整链路：导入、切章、分析、生成、校验、编辑、预览、导出。
- 产品合理性：围绕小说作者的真实改编工作流，而不是单纯聊天生成。
- 交互流畅度：Web 工作台多区域协同，用户能持续编辑和反馈。
- 创新性：YAML Schema、原文追溯、多阶段 AI、结构修复、剧本预览。

### 9.2 开发过程与质量 40%

对应设计：

- 前后端清晰分层。
- LLM provider 独立封装。
- YAML Schema 独立维护。
- 结构校验和引用校验分离。
- 明确测试用例。
- PR 和 commit 按功能拆分。

推荐 PR 拆分（与 54 小时计划的 Phase 对齐）：

1. 项目脚手架 + MockProvider + 内置样例（Phase 0）。
2. YAML Schema 校验器：结构校验 + 应用层引用/一致性校验（Phase 1）。
3. 接入真实 LLM Provider + 单阶段生成（Phase 2）。
4. 多阶段流水线（切章 / 分析 / 圣经 / 分场）+ 结构修复闭环（Phase 3）。
5. YAML 编辑器 + 剧本预览 + 导出（Phase 4）。
6. 输入拦截、测试、Demo 道具、README 与最终打磨（Phase 5）。

### 9.3 演示与表达 20%

对应设计：

- 准备内置样例小说，不依赖评委现场输入。
- 3 分钟内展示完整主流程。
- 明确展示 YAML Schema 校验通过。
- 故意制造一个 YAML 错误，展示错误定位和修复能力。
- 展示导出文件，证明结果可继续编辑。

## 10. 开发计划（54 小时 · 端到端优先）

### 10.0 排期原则

**核心原则：每个 Phase 结束都必须有一个“能从头点到尾”的可运行版本，先通后深、先丑后美。**

排期不再按模块纵切（容易出现“做了一堆模块但没集成”），而是按端到端链路横切：每个阶段都在上一个可交付版本上叠加深度。按单人 ~40 有效小时排布，显式预留 2 段睡眠与缓冲。**Phase 0–2 结束（约 17h）即拥有“丑但完整、能演示、能交”的版本，作为后续所有工作的安全垫。**

### Phase 0 ｜ 端到端骨架（0–5h）★最高优先级

目标：不接真 AI，先让“样例 → 后端 → 假 YAML → 前端显示”整条链路亮起来。

任务：

- 前后端脚手架（Vite + React + TS / Node + Express + TS），前端能 fetch 后端。
- 实现 `MockProvider`：`generate()` 直接返回写死的合法剧本 YAML。
- 准备内置样例小说（≥3 章，`examples/novel-sample.md`）——demo 命根子，现在就写好。
- 前端三块占位：小说输入区 / “生成”按钮 / YAML 显示区（先用 `<pre>`）。

验收：

- 点击“用样例生成”后，屏幕出现 YAML。
- 端到端链路打通（哪怕内容全是 mock）。

### Phase 1 ｜ 校验闭环先于真 AI（5–11h）

目标：先把校验器做出来，作为后续调 prompt、做修复、做演示的“标尺”。

任务：

- 用 `ajv` 落地 JSON Schema 草案（结构 + 枚举校验）。
- 应用层引用与一致性校验：引用指向、条件必填、ID 唯一、章节覆盖（见 5.7）。
- 打通 `POST /api/yaml/validate`，返回 `{path, message}` 数组。
- 前端校验结果面板，能列出错误路径。

验收：

- 把 mock YAML 故意改坏（删 title / 写错 speaker / 重复 id），面板能精确报出位置。

### Phase 2 ｜ 接真 LLM（单阶段先跑通）（11–17h）

目标：先用“一次调用全文 → 出 YAML”打通真实 AI 链路，暂不上多阶段。

任务：

- Provider 抽象：`MockProvider` / `OpenAIProvider` 同接口，环境变量切换，Mock 永久保留。
- 编写“全文 → YAML”单段 prompt，强约束只返回 YAML，并在 prompt 内嵌入 Schema 摘要。
- 生成结果立即过 Phase 1 校验器，返回 `{yaml, validation}`。

验收：

- 样例小说 → 真 AI → 真 YAML → 自动校验，能看到真实的校验结果（通过或具体错误）。

### 💤 睡眠 1（17–24h）

此时端到端 + 真 AI + 校验全通，已有可交付版本作为安全垫。

### Phase 3 ｜ 多阶段流水线 + 自动修复（24–33h）

目标：工程深度主战场，在稳固底座上拆分阶段。

任务：

- 切章：`/api/chapters/split`（正则识别 `第X章` / `Chapter` / Markdown 标题 / 自定义分隔符）+ 章节确认区。
- 逐章分析（摘要 / 人物 / 地点 / 事件 / 冲突）。
- 剧本圣经（统一人物表、地点表、主线），解决跨章一致性。
- 分场生成改用“章节分析 + 剧本圣经”作输入，不再喂全文（解决长文本超限）。
- 结构修复闭环（见 6.6）：校验失败 → 错误 + 原 YAML 回喂模型，只修结构不改剧情 → 再校验。

验收：

- 较长样例（建议 5 章）能稳定走完 切章 → 分析 → 圣经 → 生成 → 校验 → 修复，且修复后能转红为绿。

### Phase 4 ｜ 可编辑 + 可读预览（33–40h）

目标：落地题眼“降低门槛”，不可因赶进度裁剪。

任务：

- YAML 编辑器（Monaco 或 CodeMirror），改完可重新校验。
- 剧本预览渲染：按 beat 类型渲染为人类可读剧本（action / dialogue / narration / transition / inner_voice 各自样式）。
- 校验通过后预览自动刷新；不通过则禁用并提示。
- 导出 `.yaml`（必做）+ `.md`（可读预览，有余力则做）。

验收：

- 改 YAML → 校验 → 预览三联动顺滑；能下载 YAML。

### 💤 睡眠 2（40–47h）

### Phase 5 ｜ 打磨 + 演示 + 文档收尾（47–54h）

任务：

- “3 章以下拦截”输入校验 + 友好提示（题目硬性点）。
- 章节覆盖提示、Loading 状态、明确错误路径（不只显示“生成失败”）。
- 准备 demo 道具：预生成的合法 YAML 与故意制造的坏 YAML 各一份，存入仓库，不现场临时删字段。
- 文档收尾：`README`（介绍 / 运行 / demo 路线）、核对 `design.md` 与 `yaml-schema.md` 与代码一致。
- 录制 demo 视频 + 排练 3 分钟脚本（留最后 1–2h，不要拖到提交前）。
- 按 Phase 整理 commit / PR 说明（开发过程分）。

验收：

- 内置样例 3 分钟内完整跑通。
- 文档能解释清楚产品、架构、Schema 与设计原因。
- 代码结构清晰，主要路径有测试或可手动验收记录。

## 11. 测试计划

### 11.1 输入测试

- 3 章中文小说：允许生成。
- 2 章中文小说：阻止生成并提示至少需要 3 章。
- Markdown 标题章节：正确识别。
- 无明显章节标题：提示用户手动分隔。

### 11.2 YAML 测试

- 生成结果能被 `js-yaml` 正确解析。
- 缺少 `schema_version` 时校验失败。
- 缺少 `project.title` 时校验失败。
- `beats.type` 不在枚举中时校验失败。
- `dialogue.speaker` 指向不存在角色时校验失败。
- `dialogue`/`inner_voice` 缺少 `speaker` 时校验失败（条件必填）。
- `scene.location_id` 指向不存在地点时校验失败。
- 同类对象 `id` 重复时校验失败。
- 存在章节未被任何场景引用时给出覆盖提示。

### 11.3 预览测试

- action beat 渲染为动作段落。
- dialogue beat 渲染为角色名和对白。
- narration beat 渲染为旁白。
- transition beat 渲染为转场。
- inner_voice beat 渲染为内心独白。

### 11.4 演示测试

- 使用内置样例完成导入、生成、校验、预览、导出。
- 人为删除一个必填字段，校验面板能显示错误。
- 修复后能重新通过校验。

## 12. 风险与应对

### 12.1 AI 输出 YAML 不稳定

应对：

- 使用分阶段生成。
- prompt 中要求只返回 YAML。
- 本地解析和校验。
- 校验失败后进入修复流程。
- 全程保留 `MockProvider`：真实模型不可用时切换到 mock，主流程不中断。

### 12.2 长文本超出模型上下文

应对：

- 逐章分析。
- 将章节分析结果作为中间摘要。
- 最终生成时使用摘要和剧本圣经，而不是全文。

### 12.3 72 小时时间不足

应对：

- 优先保证 Web 工作台主流程。
- 数据库、账号、多项目管理、复杂镜头语言全部后置。
- 内置样例必须稳定可演示。

### 12.4 演示现场不可控

应对：

- 准备内置样例。
- 准备一份预生成 YAML 作为 fallback。
- 准备本地演示视频。

## 13. MVP 不包含内容

为了保证 72 小时内可交付，MVP 暂不包含：

- 用户登录。
- 多人协作。
- 云端项目管理。
- 复杂版本历史。
- Final Draft `.fdx` 导出。
- 自动分镜图生成。
- 训练或微调模型。

这些能力可以作为后续扩展方向写入 README 或答辩材料。
